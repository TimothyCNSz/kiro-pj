## 1. 项目背景与定位

- **目标**：验证「员工积分兑换」商业模式的最小可行产品（MVP）
- **用户**：公司内部员工（数百人规模），仅 PC 浏览器
- **货币**：AWSome 积分——由管理员发放/扣除，员工用于兑换
- **商品**：实物商品（需物流）+ 虚拟商品（CDK 兑换码）

> **演示/玩具定位**：真实 AWS 架构部署，但**不追求生产级性能与安全**；
> 为演示便利采用简化网络（公网可达 RDS、无 VPC Lambda）。

---

## 2. 角色与核心场景

| 角色 | 能做什么 |
|------|----------|
| **员工** | 注册/登录、浏览搜索、购物车、兑换、查看积分与兑换历史、头像 |
| **管理员** | 商品管理、积分发放/扣除（单个/批量）、发货、操作日志、低库存提醒、员工列表 |

- 统一登录入口，按角色区分权限
- **管理员只做管理，不参与兑换**（无购物车/账户内容）
- 首个管理员由部署种子预置，**不开放管理员自助注册/提权**

---

## 3. 系统架构总览

- **前端**：Vue 3 `<script setup>` + Vite + Pinia + Vue Router + vue-i18n（中/日双语）
- **后端**：Node.js + TypeScript 单体 **Lambda**（Express + serverless-express 适配 API Gateway）
- **数据层**：Drizzle ORM + postgres.js + **RDS PostgreSQL**
- **存储/分发**：S3（前端 + 上传桶）+ **CloudFront**（同一分发按路径分流）
- **邮件**：Amazon **SES**（注册验证邮件）
- **IaC**：AWS CDK（TypeScript）一键编排

---

## 3.1 AWS 部署架构

```
浏览器 ──HTTPS──▶ CloudFront
                   ├─ 默认  ─▶ S3（前端静态资源，OAC）
                   ├─ /api/* ─▶ API Gateway ─▶ 单体 Lambda ─▶ RDS PostgreSQL
                   │                                    └─▶ SES（验证邮件）
                   └─ /media/* ─▶ S3 上传桶（图片，OAC 公开读）
```

- SPA 深链接：CloudFront 把 403/404 → `/index.html`（200）
- 图片上传：后端签发**预签名 PUT URL**，浏览器直传 S3，后端不中转字节
- 演示取舍：Lambda 无 VPC 直连公网 RDS/SES，免 NAT/VPC Endpoint

---

## 4. 数据模型（核心表）

- **User / Session / EmailVerification**：账号、会话空闲过期、邮箱验证令牌
- **PointsAccount / PointsLedger**：积分余额（`CHECK balance >= 0`）+ 流水
- **Product / ProductImage / CDK**：商品、图集（主图唯一）、虚拟兑换码
- **Cart / CartItem**：服务端持久化购物车
- **Order / OrderItem**：按类型拆分的实物/虚拟订单
- **OperationLog / LowStockAlert**：操作日志、低库存去重提醒

---

## 4.1 一致性保证（不因 demo 放弃）

- **单次兑换事务原子性**：积分扣除 + 多商品库存扣减，**整体成功或整体回滚**
- **积分余额永不为负**：数据库 `CHECK` 约束兜底
- **批量扣分部分成功**：余额会变负的员工被跳过，其余正常执行
- **并发控制**：行锁 `SELECT ... FOR UPDATE` + 乐观锁版本号（演示级尽力而为）

> 这些不变式用**属性化测试（PBT, fast-check）**覆盖，488 个后端测试通过。

---

## 5. 功能流程 · 注册与邮箱验证

1. 员工用**公司邮箱**注册 → 创建「待验证」账号 + 零余额积分账户
2. 系统经 SES 发送**含验证链接/验证码**的邮件（24h 有效）
3. 点击链接 → 账号置「已激活」→ 方可登录
4. 重发验证邮件会**使旧令牌失效**，仅保留最新一枚

- 仅存令牌哈希；登录失败提示不区分「邮箱/密码错误」
- 会话空闲 **60 分钟**过期（服务端权威判定）

---

## 5.1 功能流程 · 浏览 → 加购 → 兑换

```
浏览/搜索上架商品 ─▶ 商品详情 ─▶ 加入购物车 / 立即兑换
      │                              │
      └───────────── 结算（二次确认）─┘
                        │
        含实物？──是──▶ 填写配送地址
                        │
                   积分/库存校验 ─▶ 事务扣减 ─▶ 生成订单
```

- 混合兑换按类型**拆分**实物订单与虚拟订单
- 积分不足 / 库存不足 / 缺地址 → **拒绝且无任何副作用**

---

## 5.2 功能流程 · 发货与交付

- **实物订单**：管理员上传**物流编号**（非空校验）→ 置「已发货」→ 员工可查看物流（演示假数据）
- **虚拟订单**：管理员完成**虚拟发货** → 关联并交付 **CDK** → 员工可见兑换码
- 发货前：虚拟订单**隐藏 CDK**、状态「待发货」

---

## 5.3 功能流程 · 积分与运营（管理员）

- **积分管理**：单个/批量发放或扣除；扣成负数则拒绝/跳过并提示
- **员工列表**：按邮箱搜索 + 分页，选中员工发起积分操作
- **操作日志**：商品增改、上下架、积分发放/扣除、发货，全部留痕（时间倒序）
- **低库存提醒**：库存降为 0 时**唯一触发**（去重），后台展示

---

## 5.4 功能流程 · 图片与国际化

- **图片上传**：头像 + 商品图集；JPG/PNG/WebP、≤5MB；预签名直传 S3、CloudFront 公开读
- 单商品图集 ≤5 张，主图唯一；未设头像回退默认头像
- **国际化**：中文（默认）/ 日文全量覆盖，运行时一键切换，错误码本地化映射

---

## 6. 现场演示脚本（建议顺序）

1. **员工登录** → 商品列表（本地演示图）
2. 进商品详情 → **加入购物车**（选数量）
3. 购物车 → **去结算** → 含实物**填地址** → 二次确认 → **兑换成功**
4. 查看**兑换记录 / 积分明细**
5. **切换日文**演示 i18n
6. **管理员登录** → 商品管理 / 给员工**发放积分** / **发货**（实物物流号、虚拟 CDK）
7. 查看**操作日志 / 低库存提醒**

## 7. 本地运行方式

```bat
docker compose up -d                 :: 本地 PostgreSQL
cd server
npm run build:backend
npm run db:migrate:local             :: 建表
npm run seed:local                   :: 初始管理员
npm run seed:demo:local              :: 演示商品 + 员工
npm run start:local                  :: 后端 :3000
cd .. & npm run dev                  :: 前端 :5173
```

- 本地邮件降级为**控制台打印**验证链接（`MAILER=console`）
- 前端直连后端，后端开启本地 **CORS**

## 8. 质量保障

- **属性化测试（PBT）**：fast-check 覆盖兑换原子性、余额非负、批量部分成功、
  图集不变式、令牌生命周期、图片校验等核心属性
- **单元 / 集成 / 冒烟测试**：路由权限、注册→验证→登录、事务、数据持久化
- 后端 **488 个测试通过**；前端类型检查、IaC `cdk synth` 均通过
