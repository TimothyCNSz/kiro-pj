# Requirements Document

## Introduction

AWSomeShop 是一个面向内部员工的福利电商网站，目标是通过最小可行产品（MVP）验证"员工积分兑换系统"的商业模式。员工可使用"AWSome 积分"浏览、搜索并兑换预选商品（实物或虚拟），查看积分余额与兑换历史；管理员可手动配置商品、管理员工积分、处理发货并查看简单的操作日志。

本文档定义 MVP 阶段的功能性与非功能性需求，作为后续设计与开发的契约。前端采用 Vue 技术栈，后端提供 API 与数据库以持久化数据。

### 项目定位（演示/玩具项目）
- 本项目本质上是一个**演示/玩具（demo）项目**，目标是**可演示 + 功能示意**，用于展示"员工积分兑换系统"的完整业务流程，**不追求生产级别的性能、容量与压力承载能力**。
- 尽管定位为 demo，本项目仍采用**真实的 AWS 架构**进行部署（部署形态为真），具体为：前端使用 CloudFront + S3 托管静态资源，后端使用 API Gateway + Lambda 提供接口，数据持久化使用 RDS PostgreSQL，邮件发送使用 SES。
- 由于部署平台已确定，本文档允许在需求中点名上述 AWS 服务。
- 作为为**演示便利**做出的**有意取舍**：性能需求较低、安全需求较低。为简化数据库连接与邮件发送，系统可采用**可通过公网访问的托管数据库（公网可达的 RDS）**与**可直接访问公网的无服务器计算（不置于私有 VPC 子网、可直接出网的 Lambda）**，从而无需复杂的私有网络隔离配置。此取舍不适用于生产环境。

### 范围说明
- 面向公司内部员工（数百人规模），仅支持 PC 浏览器访问，不做移动端适配。
- 注册仅限使用公司邮箱域名的邮箱，非公司邮箱域名不允许注册。
- 界面支持中文、日文双语切换，默认中文，两种语言在 MVP 中均需完整覆盖。
- 商品分为**实物商品**与**虚拟商品**两类，均在 MVP 范围内。
- 第一阶段聚焦"兑换（购买）"主流程；实物商品的物流展示数据在本阶段可使用假数据呈现。

## Glossary

| 术语 | 定义 |
|------|------|
| AWSome 积分（积分） | 平台内唯一的兑换货币，由管理员发放/扣除，员工用于兑换商品；无有效期，余额不可为负。 |
| 员工 | 已注册并登录的普通用户，可浏览、搜索、兑换商品并查看自己的余额与历史。 |
| 管理员 | 拥有后台管理权限的用户，可管理商品、积分、发货及查看操作日志。 |
| 实物商品 | 需要邮寄配送的商品，兑换后需填写配送地址，管理员通过上传物流编号发货。 |
| 虚拟商品 | 无需物流的数字商品（如 CDK/兑换码），以"虚拟发货"形式向员工提供兑换码。 |
| 购物车 | 员工暂存待兑换商品的容器，可容纳多件、多种商品，统一结算。 |
| 立即兑换 | 员工在商品详情/列表对单件商品直接发起的快捷兑换流程。 |
| 兑换订单（订单） | 一次成功兑换所生成的记录，包含商品、消耗积分、时间与状态等信息。 |
| 会话（Session） | 用户登录后建立的登录态，空闲时间超过 60 分钟后过期需重新登录。 |
| CDK | 虚拟商品的兑换码/激活码，作为虚拟发货的交付内容。 |
| 账号状态 | 账号所处的生命周期状态；新注册但尚未完成邮箱验证的账号为"待验证"，完成邮箱验证后的账号为"已激活"，仅"已激活"账号允许登录。 |
| 邮箱验证 | 员工通过点击验证邮件中的验证链接或提交验证码，证明其对所注册公司邮箱拥有访问权限的过程。 |
| 验证邮件 | 注册后系统向所注册公司邮箱发送的、包含验证链接或验证码的邮件。 |
| 验证链接/验证码 | 验证邮件中用于完成邮箱验证的凭据，自签发时刻起 24 小时内有效，过期后需重新发送。 |
| 员工头像 | 员工在个人资料中展示的头像图片，为可选项；员工未设置头像时展示默认头像。 |
| 默认头像 | 员工未上传头像时系统统一展示的占位头像图片。 |
| 商品图集 | 一件商品所关联的多张图片集合，包含一张主图与若干附图。 |
| 主图 | 商品图集中被标记为主要展示用途的图片，用于员工端列表展示。 |
| 附图 | 商品图集中除主图外的其余图片，用于商品详情的图集展示。 |
| 对象存储 | 用于存放图片文件的存储服务（S3），图片经由 CloudFront 公开读访问。 |
| 预签名上传 URL | 由系统签发、具有较短有效期的临时上传凭据，客户端凭此将图片直接上传至对象存储，后端不中转文件内容。 |
| 部署初始化（种子数据） | 在系统部署阶段执行的一次性初始化过程，用于预置基础数据；本项目用于预置首个（初始）管理员账号，此账号不经由应用内注册或提权产生。 |

## Requirements

### 需求 1：员工注册与登录

**用户故事：** 作为一名员工，我希望使用公司邮箱和密码自助注册、完成邮箱验证并登录，以便安全地访问属于我的积分和兑换功能。

#### 验收标准

1. THE 系统 SHALL 将密码强度规则定义为：密码长度至少 8 位，且至少同时包含字母与数字各一。
2. THE 系统 SHALL 将允许注册的邮箱限定为公司邮箱域名下的邮箱。
3. THE 系统 SHALL 将新注册但尚未完成邮箱验证的账号状态定义为"待验证"，将完成邮箱验证的账号状态定义为"已激活"，且仅"已激活"账号允许登录。
4. WHEN 访客提交注册表单且提供了公司邮箱域名下的有效邮箱与符合密码强度规则的密码 THEN 系统 SHALL 创建一个默认角色为"员工"、账号状态为"待验证"的账号，向该邮箱发送包含验证链接或验证码的验证邮件，并提示注册成功且需完成邮箱验证后方可登录。
5. IF 注册使用的邮箱已存在 THEN 系统 SHALL 拒绝注册并提示"该邮箱已被注册"。
6. IF 注册提交的邮箱格式无效或密码不满足密码强度规则 THEN 系统 SHALL 拒绝注册并逐项提示具体错误原因。
7. IF 注册提交的邮箱不属于允许的公司邮箱域名 THEN 系统 SHALL 拒绝注册并提示"仅支持使用公司邮箱注册"。
8. THE 系统 SHALL 将验证链接/验证码的有效期设定为自签发时刻起 24 小时。
9. WHEN 员工通过验证邮件中未过期的验证链接或提交未过期的验证码完成验证 THEN 系统 SHALL 将该账号状态更新为"已激活"并提示验证成功、可登录。
10. IF 员工使用已过期的验证链接或验证码进行验证 THEN 系统 SHALL 拒绝验证并提示"验证链接/验证码已过期，请重新发送验证邮件"。
11. WHEN 处于"待验证"状态的账号请求重新发送验证邮件 THEN 系统 SHALL 向该公司邮箱重新发送包含新的验证链接或验证码的验证邮件，并使此前签发的验证链接或验证码失效。
12. WHEN 账号状态为"已激活"的员工提交正确的邮箱与密码 THEN 系统 SHALL 校验通过、建立登录会话并跳转到商品浏览页。
13. IF 员工提交正确的邮箱与密码但账号状态为"待验证" THEN 系统 SHALL 拒绝登录并提示"邮箱尚未验证，请先完成邮箱验证"。
14. IF 登录时邮箱不存在或密码错误 THEN 系统 SHALL 拒绝登录并提示"邮箱或密码错误"，且不区分是邮箱还是密码错误。
15. WHERE 用户未登录 THE 系统 SHALL 阻止其访问需要登录的页面（浏览、兑换、余额、历史、后台）并将其重定向到登录页。

### 需求 2：会话过期管理

**用户故事：** 作为一名员工，我希望在长时间无操作后会话自动过期，以便降低账号被他人误用的风险。

#### 验收标准

1. WHEN 用户成功登录 THEN 系统 SHALL 建立空闲过期时间为 60 分钟的会话。
2. WHILE 用户会话处于有效期内 THE 系统 SHALL 允许其访问已授权的受保护资源而无需重新登录。
3. IF 用户会话空闲时间超过 60 分钟 THEN 系统 SHALL 使该会话过期。
4. WHEN 用户携带已过期会话访问受保护资源 THEN 系统 SHALL 拒绝访问并将其重定向到登录页提示重新登录。
5. WHEN 用户主动登出 THEN 系统 SHALL 立即终止当前会话。

### 需求 3：角色与权限区分

**用户故事：** 作为系统，我需要通过统一登录入口按角色区分员工与管理员，以便仅允许管理员访问后台管理功能。

#### 验收标准

1. WHERE 登录用户角色为"员工" THE 系统 SHALL 仅开放员工端功能（浏览、搜索、购物车、兑换、余额、历史）。
2. WHERE 登录用户角色为"管理员" THE 系统 SHALL 开放后台管理功能（商品管理、积分管理、发货管理、操作日志）。
3. IF 员工尝试访问任一后台管理功能或接口 THEN 系统 SHALL 拒绝访问并返回无权限提示。
4. WHEN 系统进行任何管理操作前 THE 系统 SHALL 校验请求者具备管理员权限。
5. THE 系统 SHALL 不提供管理员自助注册入口，且经由注册流程（见需求 1）创建的账号角色一律为"员工"。
6. THE 系统 SHALL 使首个（初始）管理员账号由部署初始化（种子数据）预置产生，而非通过应用内注册或提权操作产生。
7. THE 系统 SHALL 在 MVP（演示）阶段不提供将"员工"账号提升为"管理员"的应用内功能；如需新增管理员，SHALL 由部署初始化（种子数据）预置。

### 需求 4：浏览与搜索商品

**用户故事：** 作为一名员工，我希望浏览可兑换的商品列表并进行简单搜索，以便快速找到想兑换的商品。

#### 验收标准

1. WHEN 员工进入商品浏览页 THEN 系统 SHALL 展示所有处于"上架"状态的商品，每个商品至少显示名称、商品主图、所需积分与库存/可兑换状态。
2. WHERE 商品处于"下架"状态 THE 系统 SHALL 不在员工端商品列表与搜索结果中展示该商品。
3. WHEN 员工在搜索框输入关键字并提交 THEN 系统 SHALL 返回名称匹配该关键字的上架商品列表。
4. IF 搜索无匹配结果 THEN 系统 SHALL 展示"未找到相关商品"的空状态提示。
5. WHEN 员工点击某个商品 THEN 系统 SHALL 展示该商品详情，包括名称、商品图集（含主图与附图）、描述、所需积分、库存状态与商品类型（实物/虚拟）。
6. WHERE 某商品未关联任何图片 THE 系统 SHALL 在列表与详情中以占位图展示该商品。

### 需求 5：库存与积分校验

**用户故事：** 作为一名员工，我希望在积分或库存不足时得到清晰提示，以便了解为何无法兑换。

#### 验收标准

1. THE 系统 SHALL 将虚拟商品的可兑换库存等同于其可用（未使用）CDK 数量；当可用 CDK 数量为 0 时该虚拟商品视同"已兑完"。
2. IF 某商品库存为 0 THEN 系统 SHALL 在员工端将该商品标记为"已兑完"并禁止其被加入购物车或立即兑换。
3. WHEN 某商品库存因兑换降为 0 THEN 系统 SHALL 生成一条面向管理员的库存不足提醒（作为库存不足提醒的唯一触发点）。
4. IF 员工发起兑换时其可用积分小于应付积分总额 THEN 系统 SHALL 阻止兑换并提示"积分不足"。
5. IF 员工发起兑换时任一商品的请求数量超过其当前库存 THEN 系统 SHALL 阻止兑换并提示该商品库存不足。

### 需求 6：购物车管理

**用户故事：** 作为一名员工，我希望将多件、多种商品加入购物车统一兑换，以便一次性完成多个兑换。

#### 验收标准

1. WHEN 员工在上架且有库存的商品上点击"加入购物车" THEN 系统 SHALL 将该商品加入其购物车并更新购物车数量。
2. WHEN 员工在购物车中调整某商品数量 THEN 系统 SHALL 更新该商品数量并实时重算积分总额。
3. WHERE 购物车中某商品的数量超过其当前库存 THE 系统 SHALL 提示库存不足并阻止以超库存数量结算。
4. WHEN 员工从购物车移除某商品 THEN 系统 SHALL 删除该条目并重算积分总额。
5. WHEN 员工查看购物车 THEN 系统 SHALL 展示每件商品名称、单价（所需积分）、数量、小计以及应付积分总额。
6. THE 系统 SHALL 将员工购物车持久化于服务端，使其在跨会话或跨设备登录后仍然可见。

### 需求 7：兑换（结算）流程

**用户故事：** 作为一名员工，我希望通过购物车结算或对单件商品立即兑换来完成兑换，并在提交前二次确认，以便避免误操作。

#### 验收标准

1. WHEN 员工从购物车发起结算 THEN 系统 SHALL 汇总所有商品并展示应付积分总额与二次确认提示。
2. WHEN 员工对单件商品点击"立即兑换" THEN 系统 SHALL 以该单件商品发起兑换并展示二次确认提示。
3. WHERE 兑换包含实物商品 THE 系统 SHALL 要求员工在确认前填写配送地址。
4. WHEN 员工在二次确认弹窗点击确认且积分与库存校验均通过 THEN 系统 SHALL 扣除相应积分、扣减相应库存、生成兑换订单并提示兑换成功。
5. WHEN 兑换订单生成成功 THEN 系统 SHALL 从购物车移除已兑换的商品（若来自购物车）。
6. IF 员工在二次确认弹窗取消 THEN 系统 SHALL 中止本次兑换且不扣除积分、不扣减库存。
7. WHERE 兑换已成功完成 THE 系统 SHALL 不提供取消订单或退回积分的功能。
8. WHEN 系统在同一次兑换中处理多个商品时 THE 系统 SHALL 保证积分扣除与库存扣减作为一个整体成功或整体失败（不产生部分扣减）。
9. WHEN 一次结算同时包含实物商品与虚拟商品 THEN 系统 SHALL 按商品类型拆分生成独立的实物订单与虚拟订单，分别进入各自的发货流程。
10. WHERE 多名员工并发兑换同一商品 THE 系统 SHALL 在演示场景下尽力保证库存与积分扣减的一致性（尽量避免超卖与透支），但不要求提供高并发规模下的强并发控制保证。

### 需求 8：配送地址与物流查看（实物商品）

**用户故事：** 作为一名员工，我希望为实物兑换填写配送地址并查看物流状态，以便了解商品的寄送进度。

#### 验收标准

1. WHERE 兑换包含实物商品 THE 系统 SHALL 在兑换记录中保存员工填写的配送地址。
2. WHEN 管理员为某实物订单上传物流编号 THEN 系统 SHALL 将该订单状态更新为"已发货"并记录物流编号。
3. WHEN 员工查看已发货的实物订单 THEN 系统 SHALL 展示物流编号及物流状态信息（本阶段物流跟踪明细可使用假数据呈现）。
4. WHERE 订单尚未发货 THE 系统 SHALL 将该实物订单状态显示为"待发货"。

### 需求 9：虚拟商品发货（CDK）

**用户故事：** 作为一名员工，我希望兑换虚拟商品后获得兑换码（CDK），以便直接使用数字权益。

#### 验收标准

1. WHERE 兑换的商品为虚拟商品 THE 系统 SHALL 不要求填写配送地址。
2. WHEN 虚拟商品每次兑换成功 THEN 系统 SHALL 消耗该虚拟商品的 1 个未使用 CDK 作为交付内容。
3. WHERE 虚拟订单尚未完成虚拟发货 THE 系统 SHALL 将该订单状态显示为"待发货"且不展示 CDK。
4. WHEN 管理员完成虚拟发货 THEN 系统 SHALL 向员工展示对应的 CDK/兑换码并将订单状态更新为"已发货"。

### 需求 10：查看积分余额

**用户故事：** 作为一名员工，我希望查看当前积分余额，以便决定可以兑换哪些商品。

#### 验收标准

1. WHEN 员工进入积分余额页面 THEN 系统 SHALL 展示其当前可用积分余额。
2. WHEN 员工的积分因兑换、发放或扣除发生变化 THEN 系统 SHALL 在下次查看时展示更新后的余额。
3. THE 系统 SHALL 始终保证展示的员工积分余额不为负数。

### 需求 11：查看兑换历史

**用户故事：** 作为一名员工，我希望按时间查看自己的兑换历史，以便回顾过往兑换及其状态。

#### 验收标准

1. WHEN 员工进入兑换历史页面 THEN 系统 SHALL 展示其兑换订单列表，每条记录至少包含商品名称、消耗积分、兑换时间与状态（物流/发货状态）。
2. THE 系统 SHALL 将兑换历史按兑换时间从新到旧排序。
3. WHERE 兑换历史记录数量超过单页容量 THE 系统 SHALL 提供分页浏览。
4. IF 员工尚无任何兑换记录 THEN 系统 SHALL 展示空状态提示。

### 需求 12：商品管理（管理员）

**用户故事：** 作为一名管理员，我希望手动配置商品信息，以便向员工提供可兑换的商品。

#### 验收标准

1. WHEN 管理员创建商品 THEN 系统 SHALL 保存名称、商品图集、描述、所需积分、库存、上下架状态与商品类型（实物/虚拟）。
2. WHERE 商品类型为虚拟商品 THE 系统 SHALL 允许管理员为其维护用于虚拟发货的 CDK/兑换码，并将该虚拟商品的可兑换库存取值为其可用（未使用）CDK 数量。
3. WHEN 管理员编辑某商品字段并保存 THEN 系统 SHALL 更新该商品信息并对员工端后续浏览生效。
4. WHEN 管理员将商品设为"下架" THEN 系统 SHALL 使该商品不再展示于员工端且不可兑换。
5. IF 管理员提交的所需积分或库存为负数或非法值 THEN 系统 SHALL 拒绝保存并提示错误。
6. WHERE 商品类型为实物商品 THE 系统 SHALL 不强制要求 CDK 字段。
7. WHEN 管理员在商品管理中为某商品上传或管理多张图片 THEN 系统 SHALL 将这些图片关联为该商品的图集（图片上传与校验流程见需求 22）。
8. WHEN 管理员为某商品指定一张图片作为主图 THEN 系统 SHALL 将该图片标记为主图并将其余图片作为附图。
9. WHERE 某商品的图集包含至少一张图片但管理员未显式指定主图 THE 系统 SHALL 自动将其中一张图片作为主图。
10. THE 系统 SHALL 在本阶段不提供商品的物理删除功能，商品的下线（停止对员工展示与兑换）SHALL 通过将其设为"下架"实现，以避免破坏与历史订单及商品图片的关联。

### 需求 13：积分管理（管理员）

**用户故事：** 作为一名管理员，我希望向员工发放或扣除积分（可单个或批量），以便管理员工的积分权益。

#### 验收标准

1. WHEN 管理员对单个员工执行发放或扣除积分操作 THEN 系统 SHALL 相应增加或减少该员工的积分余额。
2. WHEN 管理员对多个员工执行批量发放或扣除积分操作 THEN 系统 SHALL 对所选每位员工应用相同的积分变更。
3. IF 一次扣除操作会使某员工余额变为负数 THEN 系统 SHALL 阻止该员工的此次扣除并提示余额不足。
4. WHEN 批量扣除中某些员工的余额将变为负数 THEN 系统 SHALL 跳过这些员工并提示其余额不足，同时对其余员工正常执行扣除（部分成功），并为每位实际执行扣除的员工逐一记录操作日志。
5. WHERE 管理员执行积分操作 THE 系统 SHALL 允许（但不强制要求）填写操作原因/备注。
6. WHEN 任一积分发放或扣除操作完成 THEN 系统 SHALL 记录一条操作日志。

### 需求 14：发货管理（管理员）

**用户故事：** 作为一名管理员，我希望为已兑换订单发货，以便完成实物寄送或虚拟交付。

#### 验收标准

1. WHEN 管理员为某实物订单上传物流编号 THEN 系统 SHALL 记录该物流编号并将订单状态更新为"已发货"。
2. WHEN 管理员为某虚拟订单完成虚拟发货 THEN 系统 SHALL 将对应 CDK 关联到该订单并将状态更新为"已发货"。
3. IF 管理员提交的物流编号为空 THEN 系统 SHALL 阻止将实物订单标记为已发货并提示补充物流编号。
4. WHEN 任一发货操作完成 THEN 系统 SHALL 记录一条操作日志。

### 需求 15：库存不足提醒（管理员）

**用户故事：** 作为一名管理员，我希望在商品库存耗尽时收到提醒，以便及时补充库存或下架商品。

#### 验收标准

1. THE 系统 SHALL 汇总由库存校验（见需求 5）在商品库存降为 0 时生成的库存不足提醒，而不重复触发新的提醒。
2. WHEN 管理员登录后台 THEN 系统 SHALL 展示当前库存不足的提示信息。

### 需求 16：操作日志（管理员）

**用户故事：** 作为一名管理员，我希望查看简单的操作日志，以便追溯关键的商品与积分变更。

#### 验收标准

1. WHEN 管理员执行商品增改、上下架、积分发放/扣除或发货操作 THEN 系统 SHALL 记录包含操作人、操作类型、操作对象与操作时间的日志。
2. WHEN 管理员查看操作日志 THEN 系统 SHALL 按时间从新到旧展示日志记录。

### 需求 22：图片上传与存储

**用户故事：** 作为一名用户（员工或管理员），我希望上传员工头像与商品图片并可靠地查看这些图片，以便完善个人资料并让商品展示更直观。

#### 验收标准

1. THE 系统 SHALL 支持两类图片上传对象：员工头像与商品图片。
2. THE 系统 SHALL 将允许上传的图片格式限定为 JPG、PNG 与 WebP。
3. THE 系统 SHALL 将允许上传的单张图片大小限定为不超过 5MB。
4. IF 用户上传的图片格式不属于 JPG、PNG 或 WebP THEN 系统 SHALL 拒绝该次上传并提示"仅支持 JPG、PNG、WebP 格式的图片"。
5. IF 用户上传的单张图片大小超过 5MB THEN 系统 SHALL 拒绝该次上传并提示"单张图片大小不得超过 5MB"。
6. WHEN 用户发起图片上传 THEN 系统 SHALL 签发一个具有较短有效期的预签名上传 URL，供客户端将图片直接上传至对象存储（S3）。
7. THE 系统 SHALL 使客户端凭预签名上传 URL 将图片内容直接上传至对象存储（S3），后端不中转图片文件内容。
8. IF 客户端使用已过期的预签名上传 URL 上传图片 THEN 系统 SHALL 拒绝该次上传。
9. WHEN 图片上传成功 THEN 系统 SHALL 将该图片的访问 URL 关联到对应实体（头像关联到对应员工，商品图片关联到对应商品并保留主图标记）。
10. THE 系统 SHALL 使已上传的图片经由 CloudFront 公开读访问，图片视为非敏感内容、可被公开读取与缓存。
11. THE 系统 SHALL 将单件商品可关联的图片数量上限设定为 5 张（演示级，可调整）。
12. IF 管理员为某商品上传的图片数量将超过该上限 THEN 系统 SHALL 拒绝超出部分并提示已达到每件商品的图片数量上限。

### 需求 23：个人资料与头像（员工）

**用户故事：** 作为一名员工，我希望在个人资料中上传或更换自己的头像，以便个性化我的账户展示。

#### 验收标准

1. THE 系统 SHALL 将员工头像定义为可选项，且不作为注册流程的必填内容。
2. WHERE 员工尚未设置头像 THE 系统 SHALL 展示默认头像。
3. WHEN 已登录员工在账户/个人资料处上传或更换头像 THEN 系统 SHALL 将新头像关联到该员工并在后续展示中生效（图片上传与校验流程见需求 22）。
4. WHEN 员工成功更换头像 THEN 系统 SHALL 以新头像替换此前展示的头像或默认头像。

### 需求 24：员工列表查看（管理员）

**用户故事：** 作为一名管理员，我希望查看并搜索员工列表，以便在执行单个或批量积分操作时选择目标员工。

#### 验收标准

1. WHEN 管理员进入员工列表页面 THEN 系统 SHALL 展示员工列表，每条记录至少包含邮箱、角色、账号状态与当前积分余额。
2. WHEN 管理员在员工列表输入邮箱或关键字并提交搜索 THEN 系统 SHALL 返回邮箱匹配该关键字的员工列表。
3. IF 搜索无匹配结果 THEN 系统 SHALL 展示"未找到相关员工"的空状态提示。
4. WHERE 员工列表记录数量超过单页容量 THE 系统 SHALL 提供分页浏览。
5. WHEN 管理员在员工列表中选择一个或多个员工发起积分操作 THEN 系统 SHALL 将所选员工作为目标传递至积分发放或扣除流程（见需求 13）。
6. IF 员工角色的用户尝试访问员工列表功能或接口 THEN 系统 SHALL 拒绝访问并返回无权限提示（权限约束见需求 3 与需求 20）。

## 非功能性需求 (Non-Functional Requirements)

### 需求 17：国际化（双语）

**用户故事：** 作为一名员工，我希望在中文与日文界面之间切换，以便使用我熟悉的语言。

#### 验收标准

1. THE 系统 SHALL 提供中文与日文两种界面语言，且默认显示中文。
2. WHEN 用户切换界面语言 THEN 系统 SHALL 将所有面向用户的界面文案切换为所选语言。
3. THE 系统 SHALL 在 MVP 范围内为所有面向用户的界面文案提供完整的中文与日文文案，无遗漏的未翻译文案。

### 需求 18：平台与兼容性

**用户故事：** 作为一名员工，我希望在公司 PC 浏览器上正常使用系统，以便顺利完成兑换。

#### 验收标准

1. THE 系统 SHALL 面向 PC 端主流浏览器提供正常可用的界面与功能。
2. THE 系统 SHALL 不强制要求移动端响应式适配。

### 需求 19：性能与容量（演示级）

**用户故事：** 作为一个演示项目，我只需支撑演示级别的少量用户与并发，以便完整展示业务流程，而无需生产级性能。

#### 验收标准

1. THE 系统 SHALL 支撑演示级别的少量用户与少量并发访问下的正常使用，且不要求生产级性能或压力承载能力。
2. WHEN 员工在单次兑换中处理一个或多个商品 THE 系统 SHALL 保证该次兑换的积分扣除与库存扣减作为一个整体成功或整体失败（不产生部分扣减），以保持单次兑换事务的原子一致性。
3. WHERE 数据（账号、积分、商品、订单、日志）产生变更 THE 系统 SHALL 通过后端 API 与数据库进行持久化存储。
4. WHERE 存在对同一商品的并发兑换 THE 系统 SHALL 在演示场景下尽力保证库存与积分扣减的一致性，但不要求提供高并发规模下的强并发控制保证。

### 需求 20：安全性（演示级）

**用户故事：** 作为一个演示项目，我需要保留最基本的登录鉴权与越权防护，以便展示合理的权限模型，同时不追求生产级安全加固。

#### 验收标准

1. WHERE 用户未登录 THE 系统 SHALL 拒绝访问所有受保护的数据与接口。
2. WHEN 任何积分变更发生 THE 系统 SHALL 仅允许通过受控的兑换流程或管理员操作进行，且不接受客户端直接指定余额。
3. IF 用户会话已过期 THEN 系统 SHALL 拒绝其对受保护资源的访问。
4. THE 系统 SHALL 校验每一次后台管理操作的请求者具备管理员权限。
5. THE 系统 SHALL 明确本项目为演示用途，不追求生产级安全加固；作为为演示便利做出的有意取舍，系统可接受简化的网络隔离配置（例如公网可达的 RDS 与可直接出网的 Lambda），此类取舍不适用于生产环境。

### 需求 21：界面风格

**用户故事：** 作为一名员工，我希望界面简洁易用，以便快速完成浏览与兑换。

#### 验收标准

1. THE 系统 SHALL 采用简洁通用的界面风格，无需遵循特定品牌视觉规范。
